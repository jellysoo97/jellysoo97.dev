### 웹 개발 3대장: HTML, CSS, JS

![html, css, js](/images/posts/web/big-picture-2/1.png)

웹 개발 3대장이다. HTML로 구조를 짜고, CSS로 스타일을 입히고, JS로 동작을 제어한다.

**1. HTML**

- document의 구조
  - `<head>`의 구성요소: 문자 인코딩, 문서 제목, 메타데이터, `<link>`
  - `<body>`의 구성요소: `<script>`, `defer`와 `async` attribute, HTML 태그들
- 시맨틱 HTML
  - 시맨틱 태그: 코드 블록 콘텐츠의 의미 나타내기
  - SEO
- 다양한 태그들과 태그별 속성
  - 텍스트, 목록, 링크, 테이블, 폼, 이미지, 오디오, 동영상 등
  - 태그별 고유 속성, 공통 속성, 커스텀 속성

**2. CSS**

- Cascading
  - 캐스케이딩 원리: 위에서 아래로 떨어지는, 한번에 여러 요소에 동일한 스타일 적용하기
  - 스타일 우선순위
  - 스타일 상속
- Selector 선택자
  - 태그, 클래스, id, 속성, 가상 요소
- 박스 모델
  - content box, border box
  - margin, padding
- display
  - block 요소와 inline 요소
  - flex, grid
- 반응형 디자인: media query
- 주요 속성
  - 사이즈 관련: width, max-width, min-width, height, max-height, min-height
  - 레이아웃 관련: position, z-index, overflow
  - 단위: px, em, rem, vh, vw
- CSS 트리거 리스트
  - layout, paint, composite
  - 렌더링 성능
- CSS 툴 또는 작성 방법
  - 프레임워크: TailwindCSS
  - 전처리기: Sass
  - CSS-in-JS 라이브러리: Styled Components, Emotion
  - CSS Modules: CSS 파일 모듈화

**3. JavaScript**

- ECMAScript
  - ES6
  - CommonJS, ESM
- 변수
  - 변수의 선언, 초기화, 할당
  - `var`, `let`, `const` 키워드
  - 호이스팅
  - 자바스크립트의 메모리 관리
- 데이터 타입
  - 원시 타입과 객체 타입
  - 동적 타이핑 언어, 암시적/명시적 타입 변환
- 함수
  - 함수 선언문과 표현식
  - 일급 객체
- 실행 컨텍스트
  - 코드의 평가와 실행
  - 실행 컨텍스트의 역할
- 스코프와 클로저
  - 스코프의 종류
  - 스코프 체인
  - 렉시컬 스코프
  - 클로저와 렉시컬 환경
- this 키워드
  - 호출 방식에 따른 사용
  - call, apply, bind
- 프로토타입
  - 상속과 프로토타입
- 이벤트
  - 이벤트 드리븐
  - 이벤트 객체, 브라우저 API
  - 이벤트 리스너와 이벤트 핸들러
- 비동기 처리
  - 싱글 스레드, 멀티 스레드
  - 이벤트 루프와 테스크 큐
  - cf) 블로킹, 논블로킹
- Promise
  - 비동기 처리 콜백 패턴의 한계
  - 프로미스 활용
  - async/await

### 라이브러리: React

![framework](/images/posts/web/big-picture-2/2.png)

자바스크립트만 가지고 DOM을 조작하는 건 불편하다. 반복적인 코드가 많아지고, 관리하기가 힘들다.
그래서 등장한 게 React와 같은 UI 라이브러리이다.

- 컴포넌트
  - 리액트의 작동 원리
  - 컴포넌트 생명주기
  - props와 state
  - list와 key
  - 리액트의 렌더링 방식과 가상 돔
  - 서버 컴포넌트: RSC
- 렌더링
  - 다양한 렌더링 방식: CSR, SSR, SSG, ISR
  - 렌더링 최적화
- Hooks
  - useState, useEffect
  - useMemo, useCallback, useRef
  - 커스텀 훅

### 데이터 통신과 상태관리

데이터 통신과 상태관리는 웹앱이 동작하는 데 필수요소이다.

- 데이터 통신
  - data fetch
  - axios, Tanstack Query: 데이터 패칭과 서버 데이터 상태관리
- 전역 상태관리
  - 상태관리 프로세스의 원리
  - Context API, useContext
  - Zustand, Redux, Recoil, MobX

## 웹앱을 어디에, 어떻게 올릴까

웹앱을 만들었다. 이제 웹앱을 서버에 올려야 여러 사용자들이 접근할 수 있다.
즉 모든 리소스를 서버에 배포해야 사용자들이 웹에서 접근 가능하다.

> 키워드
>
> 1. 빌드: 컴파일링, 번들링, 최적화
> 2. 클라이언트와 서버
> 3. 배포

### 빌드: 컴파일링, 번들링, 최적화

서버에 어떤 리소스를 올려야할까?
우리가 작성한 코드 파일을 그대로 서버에 올릴 수 없다.
개발자가 작성한 코드는 용량도 크고 사람에게 친화적인 코드이다.
브라우저 입장에서 이해하기 어렵다.
그래서 **빌드**한 파일을 올려야 한다.

![framework](/images/posts/web/big-picture-2/3.png)

프로젝트 빌드는 운영 환경에 최적화된 버전을 생성하는 것을 의미한다.
여러 모듈들을 최적화된 단위로 묶고(번들링), 브라우저가 알아들을 수 있는 언어로 변환(컴파일링)하고, 코드를 경량화하는 작업들을 거친다.

- 빌드 툴
  - 전처리, 컴파일, 번들링, 버전 관리 등 여러 기능 제공
  - Vite, Webpack
- 번들링
  - 코드 스플리팅
- 컴파일링
  - 컴파일링: 자바스크립트 -> 기계어로 변환, 다양한 버전의 브라우저와 호환되도록 자바스크립트 문법 변환
  - 트랜스파일링: 소스코드를 비슷한 수준의 다른 언어로 변환 ex) TypeScript -> JavaScript

### 클라이언트와 서버

서버에 올릴 빌드 파일까지 완료했다면 다음은 어떤 서버에 배포할지 생각해 볼 차례이다.
클라이언트와 서버는 역할에 따라 나뉜다.
데이터를 요청하는 쪽이 클라이언트고, 응답하는 쪽이 서버이다.
사용자=클라이언트, 구글=서버와 같이 고정된 개념이 아니다.
구글도 다른 사이트에 데이터를 요청한다면 클라이언트가 될 수 있다.

![framework](/images/posts/web/big-picture-2/4.png)

서버는 1.웹 서버와 2.웹 애플리케이션 서버(WAS)로 나뉜다.
**웹 서버**는 정적 콘텐츠 요청을 다루고, **WAS**는 동적 콘텐츠 요청+α를 처리한다.
배포하고자 하는 웹앱의 특성을 파악하여 적절한 서버 유형을 선택해야 한다.
웹 서버와 웹 애플리케이션 서버를 나눌수도 있고, 하나로 합칠수도 있고, 웹 애플리케이션 서버가 여러개일수도 있다.

- 웹 서버
  - Nginx, Apache
- 웹 애플리케이션 서버
  - Tomcat

### 배포

프로젝트 배포 구조까지 짰다면 마지막으로 배포를 해야한다.
내 로컬에 있는 빌드 파일을 원격 서버에 올려야 한다.

- 클라우드 서비스
  - AWS, Google Cloud, Naver Cloud Platform
- Docker
- 정적 웹 호스팅 서비스
  - Vercel, Netlify, AWS S3
- CI/CD 자동화
  - Jenkins, Github Actions

여분의 컴퓨터를 켜놓은채로 365일 서버를 구동시킬 게 아니라면 보통 클라우드 서비스를 이용한다.
Docker를 사용하여 이미지를 생성하고 클라우드에 컨테이너를 띄워서 배포할 수도 있다.
또한, 프론트 빌드 파일 같이 정적 콘텐츠를 배포하고 싶으면 정적 웹 호스팅 서비스를 이용하는 게 편하다.
배포 플랫폼 역시 서비스의 특성과 스팩에 맞게 선택해야 한다.

<br />

수도 없이 고쳐지는게 코드인데 그때마다 수동으로 빌드하고 배포를 하다보면 비효율적이라고 느껴질 것이다.
스크립트를 통해 빌드-(테스트)-(머지)-배포 과정을 자동화하는 게 CI/CD 자동화이다.
CI/CD 자동화를 통해 웹앱을 짧은 주기로 간편하게 업데이트할 수 있고 빌드, 배포에 투자하는 시간을 단축시켜 효율을 높일 수 있다.
