예전 JSConf 영상을 보다가 기억에 남은 말이다. 프로그램이 동작하는 원리를 알아야 동작하게끔 만들 수 있다.
사실 원리를 모르더라도 실용적인 사용법만 가지고 개발할 수 있다.
하지만 언젠가는 밑천이 드러나기 마련인 것 같다.
이참에 웹 개발의 전반적인 과정을 정리하면서 부족한 부분을 파악해서 채워나가려 한다.

## 웹이란

### 1. 인터넷

웹 전에 인터넷부터 알아보자. **인터넷**은 전 세계의 서버들을 연결하는 거대한 네트워크이다.
인터넷을 통해 우리는 데이터를 주고받고, 웹사이트에 접근하는 등 여러 서비스를 이용할 수 있다.

### 2. IP, TCP, 패킷, 데이터 링크

전 세계의 컴퓨터들이 소통하려면 통신 규약이 있어야 한다.
우리가 대화할 때 한명은 한국어로, 한명은 아랍어로 얘기하면 말이 안 통하듯이 컴퓨터간의 통신도 표준화된 규약이 필요하다.
이러한 통신 규약을 프로토콜이라고 한다.

![프로토콜](/images/posts/web/big-picture-1/2.png)

Internet Protocol Suite는 인터넷 프로토콜의 집합이다.
위 사진은 TCP/IP 4계층 모델이고 OSI 7계층 모델도 있다.
아무튼 여기서 중요한 건 TCP, IP이다.

<br />

**IP**는 발신자와 수신자가 데이터를 주고받는 데 필요한 정보 위주의 프로토콜이다.
IP의 역할은 1.수신자의 지정된 ip 주소에 2.여러 개의 작은 **패킷** 단위로 데이터를 전달하는 것이다.
그래서 ip 패킷에는 발신자의 ip, 수신자의 ip, 데이터 조각 등의 정보가 들어있다.

<br />

그런데 IP는 비연결성과 비신뢰성이라는 한계가 있다.
수신자의 상태가 메롱이어도 패킷을 전송하고(비연결성), 중간에 패킷이 사라지거나 순서가 바뀔(비신뢰성) 수도 있다.
이러한 한계를 보완하기 위한 프로토콜이 **TCP**이다.
TCP는 1.연결지향(TCP 3 way handshake) 2.데이터 전달 보증 3.순서 보장의 특징이 있다.
IP가 보장하지 못하는 비연결성, 비신뢰성을 보장하여 데이터를 안전하고 정확하게 전달하는 역할을 한다.

<br />

마지막으로 **데이터 링크**는 전선, 광섬유 등 실질적으로 데이터가 전달되는 물리 장치 간의 프로토콜이다.

### 3. WWW, HTTP

그래서 웹은 무엇일까? **WWW(World Wide Web)** 줄여서 웹은 인터넷을 통해 웹 리소스를 공유할 수 있는 시스템이다.
마찬가지로 정해진 통신 규약인 **HTTP**에 따라 원하는 웹 리소스를 요청하고 응답 받는다.

### 4. URL, DNS, 브라우저

HTTP는 많은 것을 규정하고 있는데 그 중 하나가 URL이다.
우리는 보통 **브라우저**에서 **URL**를 입력해서 웹 사이트에 접근한다.
URL은 네트워크 상에서 우리가 원하는 리소스를 식별할 수 있는 정보이다.
URL을 입력하는 것은 곧 'A 사이트 서버에 있는 B페이지 보내주세요'하고 요청하는 것과 같다.
물론 리소스는 사이트가 될 수도 있고 이미지, pdf 등 다양한 형태 일 수 있다.

<br />

요청을 보내려면 요청 수신자인 A 사이트의 ip 주소를 알아야 한다.
그런데 ip 주소는 '100.100.100.1' 이런 식으로 생겨서 기억하기 어렵다.
그래서 대부분의 웹사이트들은 도메인 네임을 **DNS**에 등록한다.
DNS는 전화번호부처럼 사람이 기억하기 쉬운 도메인 네임(ex. google.com)과 실제 ip 주소를 매핑하는 역할을 한다.

## 그래서 웹 프론트엔드 개발자는 무엇을 만들까?

![브라우저와 웹앱](/images/posts/web/big-picture-1/3.png)

웹 프론트엔드 개발자는 브라우저에서 동작하는 웹 애플리케이션(웹앱)을 만든다.
웹앱은 브라우저 위에서 동작하기 때문에 브라우저의 동작 원리를 아는 것은 도움이 된다.

> 키워드
>
> 1. 브라우저의 내부 구조
> 2. 내비게이션 과정
> 3. 브라우저의 렌더링 과정

### 브라우저의 내부 구조

![브라우저의 내부 구조](/images/posts/web/big-picture-1/4.png)

브라우저는 여러 컴포넌트들로 이루어져있다.
유저 인터페이스, 렌더링 엔진, 자바스크립트 엔진 등이 있다.
각 컴포넌트가 담당하는 작업들이 무엇인지 심화학습을 진행해보려 한다.
특히 **렌더링 엔진**은 우리가 작성한 HTML, CSS, JS 코드가 브라우저에서 어떻게 보여지는 지와 밀접한 연관이 있다.

### 내비게이션 과정

'주소창에 google.com을 입력하면 일어나는 일' < 흔히 접할 수 있는 질문이다.
네트워크를 통해 데이터를 가져올 때 브라우저가 거치는 단계를 알 수 있다.
사용자가 브라우저의 주소창에 URL을 입력하여 사이트를 요청한 후부터 브라우저가 사이트를 응답받아 렌더링을 준비하기까지의 과정을 다룬다.

### 브라우저의 렌더링 과정

브라우저의 내부 구조와 서버로부터 데이터를 가져오는 과정을 이해했다면 마지막은 가져온 리소스를 브라우저 화면에 그리는 방법이다.
브라우저가 HTML, CSS, JS를 어떻게 파싱하여 렌더링하는지 알 수 있다.
렌더링 최적화와도 관련있다.

<hr />

웹앱을 어떻게 개발하는지를 제외하고 웹 개발의 큰 흐름을 정리해봤다.
핵심은

1. 네트워크 상에서 어떻게 자원을 요청하고 응답 받는지, 어떤 프로토콜을 따르는지
2. 우리가 만드는 웹앱이 어디서 어떻게 동작하는지

를 아는 것 같다. 이어서 나머지 내용인 웹앱을 어떻게 개발하는지 정리해보고자 한다.
