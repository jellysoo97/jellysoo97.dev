웹 프론트엔드 개발의 전반적인 과정을 돌아보면서 부족한 부분이 무엇인지 되짚어보고자 한다.

**\*자세한 개발 지식을 다루는 것이 아니라 큰 흐름만 정리할 예정이다.**

## 웹 개발의 흐름

![브라우저와 웹앱](/images/posts/web/big-picture/1.png)

웹 개발은 브라우저에서 동작하는 웹 애플리케이션(웹앱)을 만들고, 유지관리하는 것이다.
HTML, CSS, JS를 배우면 어떻게든 웹앱을 만들 수 있다.
하지만 여기서 끝이 아니다.
로컬 환경에서 개발한 웹앱을 어디에 어떻게 올리는 지, 올려놓은 웹앱이 어떤 원리로 보여지는 지 등을 이해해야 웹앱을 개선해나갈 수 있다.
그러기 위해서 웹앱 자체를 살펴보는 것에 앞서 웹앱이 동작하는 환경에 대해 정리해보고자 한다.

### 웹앱은 브라우저에서 실행된다

브라우저는 사용자가 필요로 하는 리소스를 서버에 요청하고, 서버로부터 받은 응답을 화면에 보여준다.
우리는 브라우저 주소창에 URI를 입력하여 서버에 요청을 보낸다.
리소스는 주로 HTML 문서이지만 이미지, pdf 파일 등 다양한 형태일수도 있다.

<br />

브라우저와 관련해서 알아두면 좋은 내용들은 다음과 같다.

> 1. 브라우저의 내부 구조
> 2. 내비게이션 과정
> 3. 브라우저의 렌더링 과정

**1. 브라우저의 내부 구조**

브라우저는 여러 컴포넌트들로 이루어져있다.
유저 인터페이스, 렌더링 엔진, 자바스크립트 엔진 등이 있다. 각 컴포넌트가 어떤 작업을 처리하는 지 알 수 있다.

**2. 내비게이션 과정**

'주소창에 google.com을 입력하면 일어나는 일' < 흔히 접할 수 있는 질문이다.
네트워크를 통해 리소스를 가져오기 위해 브라우저가 거치는 단계를 알 수 있다.
DNS, SSL 연결 설정과 같은 개념이 등장한다.

**3. 브라우저의 렌더링 과정**

브라우저의 내부 구조와 서버로부터 리소스를 가져오는 방법을 이해했다면 마지막은 가져온 리소스를 브라우저 화면에 그리는 방법이다.
브라우저가 HTML, CSS, JS를 어떻게 파싱하여 렌더링하는지 알 수 있다.
렌더링 최적화와도 관련있다.

<hr />

### 리소스는 모두 서버에 존재한다

우리는 필요한 데이터가 있으면 항상 서버에 리소스를 요청한다.
그 말인 즉슨 모든 리소스는 서버에 있고, 서버에 리소스를 올려야 사용자들이 웹에서 접근할 수 있다.
프론트 파트에서 본다면 HTML, CSS, JS로 작성한 웹앱 파일을 서버에 올려야 여러 사용자들이 웹 사이트를 볼 수 있는 것이다.

<br />

그렇다면 어떤 리소스를 서버에 올려야 할까?

> 빌드툴
>
> 모듈 번들러: Vite, esbuild, Webpack, Rollup

우리가 작성한 HTML, CSS, JS 파일은 그대로 서버에 올리기엔 용량이 크고 여러 문제가 발생한다.
그래서 **배포 전 빌드** 과정을 거쳐야 한다. 빌드를 하는 이유는 다음과 같다.

- 코드 최적화
- 브라우저에 호환되도록 변환
- 환경에 맞는 설정 적용
- 파일 경량화

![빌드](/images/posts/web/big-picture/2.png)

빌드 결과물을 보면 내가 작성한 코드와 확연하게 다르다.
이처럼 최적화된 빌드 파일을 서버에 배포함으로써 웹앱 성능을 향상시킬 수 있다.
모든 번들러를 알아야 할 필요는 없고 사용하고자 하는 번들러만 이해해도 충분한 것 같다.

<br />

서버에 올릴 파일까지 정리했다면 다음은 어떤 서버에 어떻게 배포할지 이해할 차례이다.

![서버](/images/posts/web/big-picture/3.png)

서버의 종류는 매우 다양하다. 정확히는 서버를 구축할 수 있는 방법이 다양하다.
위 그림처럼 웹 서버와 웹 애플리케이션 서버를 나눌수도 있고, 하나로 합칠수도 있고, 웹 애플리케이션 서버가 여러개일수도 있다.
처음부터 여러 방법을 익히기보다는 서버가 어떤 역할을 하고, 어떻게 통신하는지 이해하는게 중요한 것 같다.

> 1. 클라이언트와 서버 간 통신: HTTP, SSL/TLS, SSH, DNS
> 2. 웹 서버와 웹 애플리케이션 서버: Nginx, Apache
> 3. 배포 방법: AWS EC2, Google Cloud, Docker, 정적 웹 호스팅(AWS S3, Vercel, Netlify)
> 4. CI/CD 자동화: Github Actions, Jenkins

**1. 클라이언트와 서버 간 통신**

우리가 대화를 할 때 문법에 맞는 한국어를 구사해야 말이 통하듯이 클라이언트와 서버 간 통신에도 규칙이 필요하다.
그 규칙을 프로토콜이라 한다.
HTTP는 클라이언트와 서버 간의 요청과 응답을 위한 통신 규약(프로토콜)으로, HTTP를 통해 리소스를 요청하고 응답한다.
HTTP 외에 보안과 관련된 SSL/TLS, SSH 등의 개념이 포함된다.

<br />

참고로 클라이언트와 서버는 사용자=클라이언트, 구글=서버와 같이 고정된 개념이 아니다.
구글도 다른 사이트에 데이터를 요청한다면 클라이언트가 될 수 있다.
상황에 따라 얼마든지 역할이 바뀔 수 있다.

<br />

**2. 웹 서버와 웹 애플리케이션 서버**

간략하게 정리하면 웹 서버는 정적 콘텐츠 요청을 다루고, 웹 애플리케이션 서버(WAS)는 동적 콘텐츠 요청+α를 처리한다.
하지만 앞서 말했듯이 웹 서버와 WAS가 공존하지 않을 수 있다.
배포하고자 하는 웹앱의 특성을 파악하여 적절한 서버를 선택해야 한다.
예를 들어, 블로그는 정적 콘텐츠만 다루기 때문에 웹 서버만으로 간단하게 운영할 수 있다.

<br />

**3. 배포 방법**

개발할 때 개발 서버를 구동하는 것처럼 서버가 어디에든 구동되어야만 클라이언트가 서버에 요청을 보내고 응답을 받을 수 있다.
여분의 컴퓨터를 켜놓은채로 365일 서버를 구동시킬 게 아니라면 보통 클라우드 서비스를 이용한다.
대표적으로 AWS, Google Cloud, Naver Cloud Platform 서비스가 있다.
Docker를 사용하여 이미지를 생성하고 클라우드에 컨테이너를 띄워서 배포할 수도 있다.
또한, 프론트 빌드 파일 같이 정적 콘텐츠를 배포하고 싶으면 정적 웹 호스팅 서비스를 이용하는 게 편하다.

<br />

**4. CI/CD 자동화**

수도 없이 고쳐지는게 코드인데 그때마다 수동으로 빌드하고 배포를 하다보면 비효율적이라고 느껴질 것이다.
스크립트를 통해 빌드-(테스트)-(머지)-배포 과정을 자동화하는 게 CI/CD 자동화이다.
CI/CD 자동화를 통해 웹앱을 짧은 주기로 간편하게 업데이트할 수 있고 빌드, 배포에 투자하는 시간을 단축시켜 효율을 높일 수 있다.

<hr />

웹 개발의 큰 흐름을 정리해봤다.
중간중간 이해가 부족하다고 생각되는 부분들이 있었고 추가 공부가 필요하다.
웹앱 개발 외적인 부분을 정리했으니 다음글에서는 내부로 들어가 필요한 지식들을 정리해보려고 한다.
